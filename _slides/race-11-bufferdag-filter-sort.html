Filter & sortering

Hvorfor er det relevant?

Når en webapp har dynamisk data, skal brugeren kunne finde det rigtige indhold hurtigt.

Filtrering og sortering er ofte forskellen på "data på en side" og en app, der faktisk kan bruges.
Client / server

Filtrering kan ske to steder

I en client-server-model kan vi både filtrere og sortere på serveren og på klienten.

Server-side

Supabase vælger og sorterer rækkerne, før data sendes til appen.

Client-side

React/JavaScript filtrerer eller sorterer de data, vi allerede har hentet.

I dag ser vi begge dele, men med fokus på hvordan Supabase kan gøre arbejdet allerede i requesten.

Post App

Det data vi tager udgangspunkt i

I Supabase arbejder vi med tabellen posts.

id

Unikt id. Bruges til detail view, update og delete.

created_at

Tidspunktet posten blev oprettet. God til nyeste/først eller ældste/først.

caption

Teksten under billedet. God til søgning og alfabetisk sortering.

Supabase REST

Filter sker med query parameters

Supabase bruger query parameters til at filtrere direkte i databasen.


GET /rest/v1/posts?id=eq.1

GET /rest/v1/posts?caption=ilike.*aarhus*

GET /rest/v1/posts?id=gt.5

Pointen: Vi henter kun de data, vi faktisk skal bruge.

Filter-eksempler

Hent ét bestemt post

Det her mønster bruger I allerede på detail-siden.


GET /rest/v1/posts?id=eq.4

GET /rest/v1/posts?id=eq.16

eq betyder equals: kolonnen skal være lig med værdien.

Filter-eksempler

Søg i captions

ilike betyder case-insensitive like: vi filtrerer en tekstkolonne ved at matche et mønster uden forskel på store og små bogstaver. 


GET /rest/v1/posts?caption=ilike.*aarhus*

GET /rest/v1/posts?caption=ilike.*coffee*

GET /rest/v1/posts?caption=ilike.*city*

I praksis bruger vi ofte * på begge sider, når brugeren bare skal kunne søge efter tekst, der indgår et sted i caption.

Filter-logik

Sådan læses ilike

Selve filteret er ilike. Stjernerne bestemmer, hvor i teksten matchet må ligge.

ilike.aarhus

Caption skal være præcis "aarhus", men store og små bogstaver er ligegyldige.

ilike.aarhus*

Caption skal starte med "aarhus".

ilike.*aarhus

Caption skal slutte med "aarhus".

ilike.*aarhus*

Caption skal indeholde "aarhus" et sted i teksten.

* er wildcard i URL'en og svarer til % i SQL/ILIKE.

Filter-eksempler

Filtrer på id

Id kan bruges til at hente nyere/oprettede rækker i en simpel demo.


GET /rest/v1/posts?id=gt.5

GET /rest/v1/posts?id=lt.8

GET /rest/v1/posts?id=gte.15

gt, lt og gte betyder greater than, less than og greater than or equal.

Filter-eksempler

Filtrer på dato

Dato-filter er nyttigt, når I vil vise nye posts eller indhold fra en bestemt periode.


GET /rest/v1/posts?created_at=gte.2026-04-27

GET /rest/v1/posts?created_at=lt.2026-04-24

GET /rest/v1/posts?created_at=gte.2026-04-23&created_at=lt.2026-04-24

Datoer kan filtreres med de samme operatorer som tal.

Filter-eksempler

Kombiner flere filtre

Flere query parameters sættes sammen med &.


GET /rest/v1/posts?caption=ilike.*aarhus*&id=gt.4

GET /rest/v1/posts?caption=ilike.*city*&created_at=gte.2026-04-23

GET /rest/v1/posts?caption=ilike.*aarhus*&order=created_at.desc&limit=3

Når query'en bliver lang, er URLSearchParams en god ven i React.

Operatorer

De mest brugbare i Post App

eq

Lig med. Bruges allerede i detail view: ?id=eq.1.

ilike

Case-insensitive tekstfilter. Sammen med * kan det matche præcis tekst, start, slut eller indhold.

gt / lt

Kan bruges på fx id eller datoer, hvis I vil filtrere mere specifikt.

Sortering

Sortering sker med order

order fortæller først hvilken kolonne vi sorterer på, og derefter retningen: asc eller desc. 


GET /rest/v1/posts?order=created_at.desc

GET /rest/v1/posts?order=created_at.asc

GET /rest/v1/posts?order=caption.asc

GET /rest/v1/posts?order=caption.desc

created_at.desc betyder fx: sorter efter created_at i faldende rækkefølge.

Sorterings-logik

Sådan læses asc og desc

asc

Ascending. Laveste til højeste værdi, A til Z eller ældste til nyeste.

desc

Descending. Højeste til laveste værdi, Z til A eller nyeste til ældste.

created_at.desc

Nyeste posts først, fordi de seneste datoer ligger øverst.

caption.asc

Caption sorteres alfabetisk fra A til Z.

caption.desc

Caption sorteres alfabetisk fra Z til A.

Mønsteret er: order=kolonne.retning.

Kombination

Filter, sortering og limit kan kombineres


GET /rest/v1/posts?caption=ilike.*aarhus*&order=created_at.desc&limit=5

Læs URL'en som en sætning: posts hvor caption indeholder "aarhus", nyeste først, maks 5 resultater.

HomePage

Samme fetch, men med query


const query =
  `?caption=ilike.*${searchText}*&order=${sort}`;

const response = await fetch(`${URL}${query}`, { headers });
const data = await response.json();
setPosts(data);

Det nye er ikke fetch. Det nye er URL'en.

God vane

Byg query'en mere sikkert


const params = new URLSearchParams();

if (searchText.trim()) {
  params.set("caption", `ilike.*${searchText.trim()}*`);
}

params.set("order", sort);

const response = await fetch(`${URL}?${params}`, { headers });

Det gør det lettere at tilføje flere filtre senere.

React-flow

Sådan hænger det sammen i React

1. Input Brugeren skriver "aarhus" eller vælger en sortering.
2. State searchText eller sort ændrer sig.
3. Fetch useEffect kører igen, fordi dependency array lytter på ændringer i searchText og sort.
4. UI setPosts(data) opdaterer listen på forsiden.
React-state

State holder styr på filter og sortering


export default function HomePage() {
  const [posts, setPosts] = useState([]);
  const [searchText, setSearchText] = useState("");
  const [sort, setSort] = useState("created_at.desc");

  // ...
}

Når searchText eller sort ændrer sig, kan vi hente posts på en ny måde.

useEffect

State ændrer query'en


useEffect(() => {
  async function getPosts() {
    const params = new URLSearchParams();

    if (searchText.trim()) params.set("caption", `ilike.*${searchText.trim()}*`);
    params.set("order", sort);

    const response = await fetch(`${URL}?${params}`, { headers });
    setPosts(await response.json());
  }

  getPosts();
}, [searchText, sort]);

useEffect er nyttig her, fordi dependency array får den til at køre igen, når searchText eller sort ændrer sig.

Reset

Nulstil filteret


function resetFilters() {
  setSearchText("");
  setSort("created_at.desc");
}

Når state nulstilles, kører useEffect igen og henter den oprindelige liste.

Lokalt i React

Sortering kan også ske lokalt


filteredPosts.sort((postA, postB) => {
  if (sort === "created_at.desc") {
    return new Date(postB.created_at) - new Date(postA.created_at);
  }

  if (sort === "created_at.asc") {
    return new Date(postA.created_at) - new Date(postB.created_at);
  }

  if (sort === "caption.desc") {
    return postB.caption.localeCompare(postA.caption);
  }

  return postA.caption.localeCompare(postB.caption);
});

Her sorterer vi direkte på filteredPosts, fordi filter() allerede har lavet et nyt array.

Begge dele virker

Lokal logik vs. filter i Supabase

Lokalt i React

Godt når data allerede er hentet, og listen er lille eller mellemstor.

I Supabase query

Godt når databasen selv skal finde de rigtige rækker, før de sendes til appen.

Fordel lokalt

Nemt at forstå, hurtigt at bygge og godt til tidlige øvelser i React/JavaScript.

Fordel i backend

Færre data over netværket og mere præcise requests, især når tabellen vokser.

Pointen er ikke enten/eller. Pointen er at vælge det sted, der giver mest mening for opgaven.
